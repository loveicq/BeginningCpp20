### 第2页

**从源代码到可执行文件的翻译**

从一组C++源代码文件创建可执行文件基本上是一个三阶段的过程：预处理、编译和链接。

首先，预处理器根据你通过预处理指令指定的说明，准备和修改每个源文件。它递归地分析、执行，然后从源代码中移除所有这些指令。预处理器的输出称为**翻译单元**，它完全由C++声明和语句组成。

接下来，编译器独立处理每个翻译单元以生成一个**目标文件**。目标文件包含它在翻译单元中遇到的定义的机器码，以及对所谓的**外部实体**的引用——这些实体被翻译单元使用但未在其中定义。外部实体的例子包括由导入的模块定义和导出的实体。在本附录后面，你也会遇到其他外部实体的例子。

最后，**链接器**组合程序的所有目标文件以生成最终的可执行文件。它建立所有源文件的所有目标文件之间的所有必要连接。如果一个目标文件包含对某个外部实体的引用，但在任何其他目标文件中都找不到该实体，则不会生成可执行文件，并且链接器会报出一个或多个错误消息。

预处理源文件、编译翻译单元和链接目标文件的组合过程称为**翻译**。

> **注意**：为了方便，预处理和编译阶段通常由同一个可执行文件（通常称为“编译器”）执行。但是，大多数编译器系统也允许你根据需要单独调用预处理器。

尽管C++20中模块的引入（见第11章）对我们今天将源代码翻译成可执行文件的方式产生了重大影响，但整体的三阶段过程大致保持不变。所有源文件——包括模块文件——仍然被预处理成翻译单元，然后编译成二进制目标文件，最后所有这些文件被链接在一起形成一个可执行文件。最大的区别在于，使用模块时，预处理器不再将整个模块接口的源代码文本复制粘贴到每个导入它的翻译单元中。相反，如第11章所述，编译器依赖于模块接口的预编译二进制文件来处理任何使用它的翻译单元。本附录后面会详细介绍包含头文件和使用模块之间的区别。

不过，在我们能正确解释这些区别之前，我们首先需要准确解释头文件包含是如何工作的。为此，你需要了解预处理是如何工作的，以及如何使用不同的预处理指令来控制它。

**预处理指令**

所有预处理指令都以符号 `#` 开头，因此很容易与C++语言声明区分开。表 A-1 显示了完整的指令集。

预处理是一种源自C语言的底层机制，在现代C++中变得越来越不重要。C++的能力通常提供了更有效和更安全的方式来达到与表 A-1 中预处理指令相同的结果。例如，模块提供了一种远比使用 `#include` 指令更令人满意的将代码组织到多个自包含文件中的方式。而常量和函数模板远比使用 `#define` 指令定义的宏更安全、更强大。在本附录中，我们将指出C++提供的这些和其他替代方案。

尽管如此，大多数预处理指令在现代C++中仍然有其用途，尽管只是偶尔。例如，当你的代码必须能使用不同的编译器编译，或者你的目标是多个操作系统时，偶尔的条件预处理块往往是不可避免的。当然，你几乎在任何遗留的头文件或源文件中仍然会遇到预处理指令。所以你肯定仍然至少需要了解不同预处理指令的基本作用。

我们首先解释如何使用 `#define` 指令定义宏。

**定义预处理器宏**

`#define` 指令指定一个所谓的**宏**。宏是一个文本重写规则，它指示预处理器在将源代码交给编译器之前对其应用哪些文本替换。我们区分两种类型的宏：**类对象宏**和**类函数宏**。

### 第4页

**定义类对象宏**

`#define` 预处理指令的最简单形式如下：

```cpp
#define IDENTIFIER sequence of characters
```

这个所谓的**类对象宏**有效地将 `IDENTIFIER` 定义为 `sequence of characters` 的别名。`IDENTIFIER` 必须符合C++中标识符的通常定义。`sequence of characters` 可以是任何字符序列，包括空序列或包含空格的序列。事实上，替换序列为空的类对象宏相对常见。

`#define` 最明显的用途无疑是指定某个标识符，在预处理期间将其替换为非空的替代字符串。例如，你可以如下定义 `PI` 作为表示数值的字符序列的别名：

```cpp
#define PI 3.1415
```

尽管这个 `PI` 宏看起来像一个变量，但它与变量毫无关系。`PI` 是一个符号，或称为标记，在代码编译前由预处理器将其替换为指定的字符序列。对预处理器而言，`3.1415` 不是一个数值。它仅仅是文本。一个字符串。预处理期间不会验证这个序列是否构成一个有效的数字。预处理器盲目地替换文本，对其右侧的字符序列没有任何安全检查。

此外，如果你的代码包含任何名为 `PI` 的C++实体（变量、函数、类型等），预处理器会盲目地将该标识符的所有出现替换为 `3.1415`。假设你不知道 `PI` 宏，并写了下面这个原本正确的C++语句：

```cpp
const double PI = std::numbers::pi;
```

那么预处理器只看到一个标记 `PI`，它已被指示用字符序列 `3.1415` 替换。预处理器既不知道也不关心这个标记出现在C++赋值的左侧。它只是用文本替换标记。因此，它交给编译器的是：

```cpp
const double 3.1415 = std::numbers::pi;
```

同样，如果宏被称为 `pi` 而不是 `PI`，预处理器也会同样乐意地将你的代码重写成这个同样无意义的语句。

```cpp
const double PI = std::numbers::3.1415;
```

无论哪种情况，编译器都会产生一系列令人费解的错误消息。

尽管 `#define` 指令在C语言中常用于定义诸如 `PI` 这样的符号常量，但现在你应该清楚为什么你永远、永远不应该在C++中这样做。定义常量变量要好得多、安全得多：

```cpp
const double PI { 3.14159265358979323846 };
```

### 第5页

> **警告**：当 `const` 变量（或变量模板）能解决问题时，绝对不要在C++中使用宏。使用 `#define` 指令至少有三大缺点：没有类型检查支持，不遵守作用域，并且标识符名称不能绑定到命名空间中。

不过，预处理器并非完全对C++语法一无所知。它只在标识符是标记时才进行替换；如果标识符是更长标识符、字符串字面量或代码注释的一部分，则永远不会替换。

这里有一个使用类对象宏定义的程序示例：

```cpp
// ExA_01.cpp
// 定义类对象宏
import <iostream>;

#define POINTER_SIZE sizeof(int*)*BITS_PER_BYTE
#define BITS_PER_BYTE 8

int main() {
  std::cout << POINTER_SIZE << std::endl; // 通常是 32 或 64
}

// 这个宏（虽然为"main"提供了一个完全有效的定义）永远不会被应用，
// 因为当预处理器处理"int main()"这行时它还没有被定义。
#define main Of chief or leading importance; prime, principal.
```

使用这些宏，预处理器首先将任何出现的 `POINTER_SIZE` 替换为字符序列 `sizeof(int*)*BITS_PER_BYTE`，然后进一步将其重写为 `sizeof(int*)*8`。也就是说，展开一个宏后，预处理器会从头开始重新处理得到的字符序列，在需要的地方替换更多的标记。这个过程一直持续到没有更多标记需要替换为止。

如第2章所述，`sizeof` 运算符计算出用于表示一个类型或变量所需的字节数。在 ExA_01 中我们任意选择了 `int*` 类型，但任何其他指针类型也可以。对于32位程序，任何指针占用4字节或32位；对于64位程序，任何指针占用8字节或64位。因此 ExA_01 的输出通常显示为32或64。

预处理器从上到下处理文件，沿途执行指令和进行文本替换。这就是为什么当预处理器到达 `int main()` 这一行时，还没有定义标识符为 `main` 的宏。这个宏直到几行之后才被定义，因此永远不会被应用。

### 第6页

另请注意，我们有意在 `POINTER_SIZE` 宏之后定义了 `BITS_PER_BYTE` 宏，以说明文本替换发生的顺序。也就是说，预处理器在执行 `#define` 指令时并不处理宏定义的右侧；它只在每次宏展开后才这样做。

**定义空宏**

对替换标识符的字符序列没有限制。它甚至可以不存在，在这种情况下，标识符存在但没有预定义的替换字符串——替换字符串为空。如果你不为标识符指定替换字符串，那么代码中该标识符的出现将被替换为空字符串；换句话说，该标识符将被移除。例如：

```cpp
#define VALUE
```

结果是，该指令之后所有出现的 `VALUE` 都将被移除。

然而，更多时候，你定义空的类对象宏并不是为了执行文本替换。在我们的例子中，该指令也将 `VALUE` 定义为一个标识符，其存在性可以被其他指令测试，然后用于条件编译。在标准C++中，这种技术最常见的单一用途可能是头文件的管理，正如你将在本附录末尾看到的那样。另一个典型用途是基于编译器或构建系统设置的宏进行条件编译。例如，这样的宏可能指示目标平台的属性（如操作系统、地址大小等），或某些编译器设置（如调试构建的切换）。在简要介绍类函数宏之后，你将很快看到条件编译的例子。

**定义类函数宏**

除了类对象宏，你还可以使用 `#define` 指令定义类函数的文本替换宏。例如：

```cpp
#define MAX(A, B) A >= B ? A : B
```

虽然这个宏看起来很像一个函数，但它肯定不是。没有参数类型，也没有返回值。宏不是被调用的东西，它的右侧也不一定指定在运行时执行的语句。我们的示例宏仅仅指示预处理器将源代码中所有出现的 `MAX(anything1, anything2)` 替换为 `#define` 指令后半部分出现的字符序列。在这个替换过程中，宏参数标识符 `A` 在 `A >= B ? A : B` 中的所有出现当然会被替换为 `anything1`，所有 `B` 的出现会被替换为 `anything2`。预处理器不尝试解释或评估 `anything1` 和 `anything2` 字符序列；它所做的又仅仅是盲目的文本替换。例如，假设你的代码包含这个语句：

```cpp
std::cout << MAX(expensive_computation(), 0) << std::endl;
```

那么预处理器会在将其交给编译器之前将其展开为以下源代码：

```cpp
std::cout << expensive_computation() >= 0 ? expensive_computation() : 0 << std::endl;
```

### 第7页

这个例子暴露了两个问题：

1.  结果代码将无法编译。如果你将三元运算符 `?:` 与流操作符 `<<` 一起使用，C++的运算符优先级规则告诉我们，包含三元运算符的表达式应该放在括号内。因此，更好地定义 `MAX` 宏的方法如下：

    ```cpp
    #define MAX(A,B) ((A) >= (B) ? (A) : (B))
    ```

2.  `expensive_computation()` 函数最多被调用两次。如果一个宏参数（如 `A`）在替换文本中出现不止一次，预处理器将盲目地多次复制宏实参。这种宏的不良行为不仅难以避免，也更难发现。

这些只是宏定义中常见陷阱的两个例子。因此，我们建议你只有在有充分理由时才创建类函数宏。虽然某些高级场景确实需要宏，但C++大多提供了更优越的替代方案。宏在C程序员中很流行，因为它们允许创建适用于任何参数类型的类函数结构。但你已经知道C++为此提供了更好的解决方案：函数模板。在第10章之后，对你来说定义一个替代C风格 `MAX` 宏的C++函数模板应该是轻而易举的事。C++函数模板本质上避免了前面列出的宏定义的两个缺点。

> **警告**：绝不要使用预处理器宏来完成函数模板能完成的工作。相反，你应该始终使用（重载的）C++函数或函数模板。函数模板在定义适用于任何参数类型的函数蓝图方面远远优于预处理器宏。C++函数是类型安全的，遵守作用域，可以被重载，它们的名字可以绑定到命名空间，并且比宏更容易调试。此外，函数实参永远不会被求值超过一次。

> **注意**：由于宏的上述所有缺点，宏甚至无法从C++20模块中导出。这是在现代C++中总是定义适当的变量、函数和模板而不是宏的另一个可靠理由。

### 第8页

**预处理器运算符**

为了完整起见，表 A-2 列出了可以应用于类函数宏参数的两个运算符。

**表 A-2. 预处理器运算符**

| 运算符 | 描述                                                               |
| :----- | :----------------------------------------------------------------- |
| `#`    | 字符串化运算符。将实参转换为包含其值的字符串字面量。               |
| `##`   | 连接运算符。连接两个标识符的值。                |

下面的示例程序说明了如何使用这些运算符：

```cpp
// ExA_02.cpp
// 使用预处理器运算符
import <iostream>;

#define DEFINE_PRINT_FUNCTION(NAME, COUNT, VALUE) \
 void NAME##COUNT() { std::cout << #VALUE << std::endl; }

DEFINE_PRINT_FUNCTION(fun, 123, Test 1 "2" 3)

int main() {
  fun123();
}
```

在讨论两个预处理器运算符的用法之前，ExA_02.cpp 展示了另一件事：宏定义通常不允许跨越多行。预处理器默认会用它在标识符右侧同一行找到的所有字符替换宏标识符的任何出现。然而，将整个定义放在一行并不总是实用的。因此，预处理器允许你添加换行符，只要它们前面紧跟着一个反斜杠字符。所有这些转义的换行符都会从替换文本中丢弃。

在 ExA_02.cpp 中，我们在宏定义的右侧之前添加了换行符，这可能是最自然的做法。但由于预处理器总是将任何被分割的行拼接回来，而不解释这些字符，这种转义换行符实际上可以出现在任何你想要的地方。这并不是推荐的，但理论上你甚至可以写出以下完全有效的C++程序（等同于 ExA_02.cpp）：

```cpp
import <iostream>;

#define DEFINE_PRINT_FUNCTION(NAME, COUNT, VALUE) vo\
 id NAME##COUNT() { std::co\
 ut << #VALUE << std::endl; }
```

关于多行宏和换行符就讲这么多；是时候回到手头的话题了：预处理器运算符。ExA_02 中的宏定义同时使用了 `##` 和 `#`：

void fun123() { std::cout << "Test 1 \"2\" 3" << std::endl; }
```

标记 `fun` 和 `123` 通过 `##` 连接。如果没有 `##` 运算符，你只能在 `NAMECOUNT` 或 `NAME COUNT` 之间选择。对于前者，预处理器不会识别 `NAME` 或 `COUNT`；对于后者，我们的示例宏将展开为 `fun 123`，这不是一个有效的函数名（C++标识符不能包含空格）。

如果没有 `#` 运算符，那么将 `VALUE` 的值转换为字符串字面量的最佳尝试是 `"VALUE"`，但这因两个原因而不起作用。首先，`"VALUE"` 只会产生字符串字面量 `"VALUE"`，因为预处理器再次不会替换字面量、标识符或代码注释内的宏参数标识符。其次，即使在更复杂的情况下，也没有办法注入所需的转义字符。

因为预处理器首先运行，所以当C++编译器处理代码时，`fun123()` 函数的定义已经存在。这就是为什么你可以在程序的 `main()` 函数中调用 `fun123()`，它会产生以下结果：

```
Test 1 "2" 3
```

因此，ExA_02 说明了你绝对不限于用宏指定仅变量或类函数的结构。也就是说，你还可以使用宏生成整个函数定义，甚至整个类定义。

### 第10页

**取消定义宏**

你可能希望 `#define` 指令产生的宏仅应用于程序文件的某一部分。然而，将宏定义放在由花括号 `{` 和 `}` 界定的作用域内是无效的。预处理宏不遵守作用域，如下例所示：

```cpp
{
#define PI 3.141592653589793238462643383279502884
//...
} // 从此之后代码中所有出现的 PI 仍然会被替换！
```

但是，你可以使用 `#undef` 指令取消一个标识符的定义。你可以用这个指令否定先前定义的宏：

```cpp
#undef IDENTIFIER
```

在该指令之后，`IDENTIFIER` 不再被定义，因此在该行之后不再发生对 `IDENTIFIER` 的替换。以下代码片段说明了这一点：

```cpp
#define PI 3.141592653589793238462643383279502884

//...

#undef PI

// 从此处开始，PI 不再被定义，因此不会发生替换。
// 代码中任何对 PI 的引用都将保持不变。
```

在 `#define` 和 `#undef` 指令之间，预处理将代码中适当的 `PI` 出现替换为 `3.141592653589793238462643383279502884`。在其他地方，`PI` 的出现保持原样。`#undef` 指令也适用于类函数宏。例如：

```cpp
#undef MAX
```

> **提示**：Windows 操作系统的本地 API 函数通过诸如 `<windows.h>` 这样的头文件提供。这个头文件的一个众所周知的烦人之处在于，默认情况下，它们定义了类似于我们之前的 `MAX()` 宏的 `min()` 和 `max()` 预处理宏。首先，这些宏当然遭受与任何类函数宏相同的问题：与函数不同，它们的参数可能被求值两次，这可能会无意中导致性能下降。其次，Windows 头文件注入的 `min()` 和 `max()` 宏会使原本完全正确的C++表达式（如 `std::max(o, my_val)`）失效，因为它们指示预处理器将这些表达式重写为无意义的文本。
>
> 一种解决方案是在包含这些 Windows 头文件后取消定义这两个宏，如下所示：
> ```cpp
> #include <windows.h>
> #undef min
> #undef max
> ```

### 第11页

> 第三种，在这种情况下不太优雅的解决方案是求助于诸如 `(std::max)(o, my_value)` 这样的表达式。通过在 `std::max` 周围添加一对额外的括号，你可以在预处理器眼中将这个限定名变成一个单一的标记，从而阻止不需要的宏展开。

### 第12页

类似地，你可以使用 `#if !defined` 或其等效形式 `#ifndef` 来测试一个标识符是否未被定义：

```cpp
#if !defined MY_IDENTIFIER
// 如果 MY_IDENTIFIER 没有被定义，则直到 #endif 的代码将被放入源文件中。
// 否则，该代码将被省略。
#endif
```

假设你将以下代码放在你的程序文件中：

```cpp
// 设置数组 data[] 的代码...

#ifdef CALC_AVERAGE
  double average {};
  for (size_t i {}; i < std::size(data); ++i)
    average += data[i];
  average /= std::size(data);
  std::cout << "Average of data array is " << average << std::endl;
#endif

// 程序的其余部分...
```

如果 `CALC_AVERAGE` 标识符已被先前的预处理指令定义，那么 `#if` 和 `#endif` 指令之间的代码将作为程序的一部分被编译。如果 `CALC_AVERAGE` 未被定义，则该代码不会被包含。

因为编译器甚至看不到条件块中条件不满足部分的代码，这些代码甚至可以包含它根本不知道如何编译的语句。这就是为什么这种技术常被用来根据仅为特定编译器定义的宏，或在针对特定操作系统、处理器类型等时进行条件编译。

除了测试给定的宏是否被定义，你当然可以更一般地使用 `#if` 指令来测试任何常量表达式是否为真。让我们进一步探讨一下。

**测试特定的标识符值**

`#if` 指令的一般形式如下：

```cpp
#if constant expression
```

`constant expression` 必须是一个不包含强制类型转换的整型常量表达式。如果 `constant expression` 的值非零，则 `#if` 之后直到 `#endif` 的行将被包含在要编译的代码中。最常见的应用是使用简单的比较来检查特定的标识符值。例如，你可能有以下语句序列：

```cpp
#if ADDR == 64
  // 利用64位寻址的代码...
#endif
```

### 第13页

> **提示**：没有跨平台的宏标识符来检测当前目标平台是否使用64位寻址。然而，大多数编译器确实提供一些特定于平台的宏，当它针对64位平台时，会为你定义这些宏。例如，一个应该在 Visual C++、GCC 和 Clang 编译器上工作的具体测试看起来像这样：
> ```cpp
> #if defined(WIN64) || defined(__x86_64__) || defined(__ppc64__)
>   // 利用64位寻址的代码...
> #endif
> ```
> 请查阅你的编译器文档以了解这些和其他预定义的宏标识符。

**多选代码选择**

`#else` 指令的工作方式与C++的 `else` 语句相同，它在 `#if` 条件失败时标识要包含在文件中的行序列。这提供了两个块的选择，其中一个将被合并到最终的源代码中。例如：

```cpp
#if ADDR == 64
  std::cout << "Standard 64-bit addressing version." << std::endl;
  // 利用64位寻址的代码...
#else
  std::cout << "Legacy 32-bit addressing version." << std::endl;
  // 用于旧式32位处理器的代码...
#endif
```

根据 `ADDR` 是否被定义为 `64`，这两个语句序列中的一个将被包含在文件中。

有一种特殊形式的 `#if` 用于多选选择。这就是 `#elif` 指令，它的一般形式如下：

```cpp
#elif constant expression
```

`#elif` 指令的好处在于，与几乎等价的 `#else #if` 结构不同，你仍然只需要一个 `#endif` 来结束条件编译块。以下是你可能如何使用它的示例：

```cpp
#if LANGUAGE == ENGLISH
  #define Greeting "Good Morning."
#elif LANGUAGE == GERMAN
  #define Greeting "Guten Tag."
#elif LANGUAGE == FRENCH
  #define Greeting "Bonjour."
```

### 第14页

```cpp
#else
  #define Greeting "Ola."
#endif

std::cout << Greeting << std::endl;
```

通过这一系列指令，根据先前 `#define` 指令中分配给 `LANGUAGE` 的值，输出语句将显示几种不同问候语中的一种。

> **警告**：在条件指令 `#if` 和 `#elif` 之后出现的任何未定义标识符都将被替换为数字 `0`。这意味着，如果在前面的例子中 `LANGUAGE` 没有被定义，那么如果 `ENGLISH`、`GERMAN` 或 `FRENCH` 也没有被定义，或者被显式定义为零，它仍然可能等于它们。

另一个可能的用途是根据表示版本号的标识符包含不同的代码：

```cpp
#if VERSION == 3
  // 版本3的代码...
#elif VERSION == 2
  // 版本2的代码...
#else
  // 原始版本1的代码...
#endif
```

这允许你维护一个单一的源文件，根据如何在 `#define` 指令中设置 `VERSION`，该文件可以编译生成程序的不同版本。

> **提示**：你的编译器可能允许你通过向编译器传递命令行参数来指定预处理标识符的值。这样，你可以在不更改任何代码的情况下编译同一程序的不同版本或配置。

**测试可用头文件**

标准库的每个版本都提供了大量的新头文件。一方面，你通常总是想使用C++所能提供的最好和最新的功能。但另一方面，你的代码有时应该能够使用多个编译器正确编译和运行。这有时需要一种方法，让你在编译时测试当前编译器可以访问哪些头文件，以便启用或禁用代码的不同版本。

### 第15页

`__has_include()` 宏可用于检查任何头文件的可用性，无论是标准库的一部分、某个其他外部库的一部分，还是你自己的头文件之一。以下是一个示例：

```cpp
#if __has_include(<execution>)
  #include <algorithm>
  #include <execution>
  // ... 使用 std::for_each(std::execution::par, ...) 的定义
#elif __has_include(<tbb/parallel_for.h>)
  #include <tbb/parallel_for.h>
  // ... 使用 Intel TBB 库中 tbb::parallel_for(...) 的定义
#else
  #error("Support for parallel loops is required for acceptable performance")
  // ... 提供使用较慢的顺序 for 循环的替代定义
#endif
```

我们相信从这个示例大纲中你能弄清楚它是如何工作的。

**测试可用语言特性**

通过上一节的 `__has_include()` 宏，你可以检查某个标准库头文件是否存在，但它不会告诉你该头文件内容的任何信息。新版本的C++也经常向现有头文件添加功能。例如，C++20 将 `unseq` 执行策略对象添加到了 C++17 的 `<execution>` 头文件中。之前的 `__has_include(<execution>)` 条件无法告诉你这个新策略是否被支持。其他例子包括扩展 `std::make_shared()` 以支持创建共享数组，以及向 `<math>` 添加 `std::lerp()` 和 `std::midpoint()` 线性插值函数。你如何知道你的代码是否可以利用这些特性，或者是否应该回退到可能效率较低的替代方案？

从C++20开始，你可以使用 `<version>` 头文件中定义的宏来测试你的编译器是否支持特定的标准库特性。以下是你如何测试我们在前一段中提到的不同特性：

```cpp
#ifdef __cpp_lib_execution               // <execution> 头文件存在 (C++17)
#if __cpp_lib_execution >= 201902L       // <execution> 定义了 unseq 策略 (C++20)
#ifdef __cpp_lib_shared_ptr_arrays       // std::shared_ptr<> 支持数组 (C++17)
#if __cpp_lib_shared_ptr_arrays >= 201707L // std::make_shared<>() 支持数组 (C++20)
#ifdef __cpp_lib_interpolate             // std::lerp() 和 midpoint() 添加到 <math> (C++20)
#if __cpp_lib_interpolate >= 201902L     // 等同于上一个 (201902L 是初始值)
```

### 第16页

一旦标准库实现完全支持一个新的库特性，相应的特性测试宏要么被定义，要么被更新。从那时起，该宏将展开为一个整数，表示该特性正式添加到C++标准的月份。每当该特性有重大更新时，这个数字就会增加。

`<version>` 头文件定义了超过150个这样的库特性测试宏。如果需要知道有哪些宏可用以及它们可能的值，你可以查阅标准库参考。

当然，不仅仅是标准库在发展，C++语言本身也在发展。因此，存在类似的宏来测试你的编译器是否已经支持某个语言特性。例如，如果编译器支持 lambda 表达式，则定义 `__cpp_lambdas`；如果支持概念，则定义 `__cpp_concepts`，等等。这些语言特性测试宏在每个翻译单元中都被隐式定义，无需包含任何头文件或导入任何模块。

大多数特性测试宏是类对象的。唯一的例外是 `__has_cpp_attribute()`。你可以使用这个类函数宏来测试你的编译器是否支持给定的C++属性。以下是你如何定义一个 `FALLTHROUGH` 宏，如果编译器已知支持 `[[fallthrough]]`，则该宏展开为 `[[fallthrough]];`；否则，该宏被定义为空。

```cpp
#ifdef __has_cpp_attribute
  #if __has_cpp_attribute(fallthrough)
    #define FALLTHROUGH [[fallthrough]];
  #else
    #define FALLTHROUGH
  #endif
#else
  #define FALLTHROUGH
#endif
```

请注意，为了与旧编译器完全兼容，我们必须首先检查 `__has_cpp_attribute` 宏本身是否被定义。如果没有这个宏，就没有可靠的、标准化的方法来检查你的编译器是否支持 `[[fallthrough]]` 属性。

> **注意**：尽管特性测试宏直到C++20才被添加到官方标准中，但几个主要编译器从大约2013年就开始支持这些宏。不幸的是，当时并非所有主要编译器都效仿，因此为了与所有遗留编译器完全兼容，你可能暂时仍需依赖非标准的方法。

### 第17页

**标准预处理器宏**

除了上一节讨论的特性测试宏之外，还有几个标准的预定义预处理器宏。表 A-3 列出了一些有用的例子。

**表 A-3. 预定义的预处理器宏**

| 宏              | 描述                                                                        |
| :-------------- | :-------------------------------------------------------------------------- |
| `__DATE__`      | 源文件被预处理时的日期，格式为 `Mmm dd yyyy` 的字符串字面量。               |
| `__TIME__`      | 源文件被编译时的时间，格式为 `hh:mm:ss` 的字符串字面量。              |
| `__LINE__`      | 当前源代码行的行号，十进制整数字面量。                               |
| `__FILE__`      | 源文件的名称，字符串字面量（被 `<source_location>` 取代）。               |
| `__func__`      | 当前函数的名称，字符串字面量（被 `<source_location>` 取代）。               |
| `__cplusplus`   | 一个 `long` 类型的数字，对应于你的编译器支持的最高C++标准版本。格式为 `yyyymm`。 |

请注意，表 A-3 中的每个宏名称都以两个下划线字符开头，大多数也以两个下划线字符结尾。你可以使用日期和时间宏来记录程序上次编译的时间，例如：

```cpp
std::cout << "Program last compiled at " << __TIME__ << " on " << __DATE__ << std::endl;
```

当这个语句被编译时，显示的值是固定的，直到你再次编译它。因此，程序输出其上次编译的时间和日期。这些宏对于在“关于”屏幕或日志文件中使用很有用。

`__LINE__` 和 `__FILE__` 宏展开为与源文件相关的参考信息。你可以使用 `#line` 指令修改当前行号，随后的行号将从那里开始递增。例如，要从1000开始行号，你可以添加这个指令：

```cpp
#line 1000
```

### 第18页

你可以使用 `#line` 指令来更改 `__FILE__` 宏返回的字符串。它通常产生完全限定的文件名，但你可以将其更改为你喜欢的任何内容。例如：

```cpp
#line 1000 "The program file"
```

这个指令将下一行的行号更改为1000，并将 `__FILE__` 宏返回的字符串更改为 `"The program file"`。这不会更改文件名，只会更改宏返回的字符串。当然，如果你只想更改显示的文件名而保持行号不变，你能做的最好的事情是在 `#line` 指令中使用 `__LINE__` 宏：

```cpp
#line __LINE__ "The program file"
```

你可能想知道为什么你会想使用 `#line` 指令来更改行号和/或文件名。这种需求很少见，但一个例子是将某种其他语言映射到 C 或 C++ 的程序。原始语言的一条语句可能会生成多条 C++ 语句，通过使用 `#line` 指令，你可以确保 C++ 编译器错误消息标识原始代码中的行号，而不是生成的 C++ 代码中的行号。这使得更容易识别原始代码中导致错误的语句。

> **提示**：C++20 的 `<source_location>` 模块提供了比 `__LINE__` 和 `__FILE__` 宏更好、更强大的替代方案。以下片段展示了一个典型用法：
> ```cpp
> void logError(std::string_view errorMessage, std::source_location location = std::source_location::current()) {
>   std::cerr << std::format("{}:{}:{} - An unexpected error occurred in {}: {}",
>                            location.file_name(), location.line(), location.column(),
>                            location.function_name(), errorMessage) << std::endl;
> }
> ```
> 另一个优点是你可以轻松存储 `std::source_location` 对象并传递它们。

### 第19页

**使用 `assert()` 宏**

`assert()` 预处理宏在标准库头文件 `<cassert>` 中定义。它使你能够测试程序中的逻辑表达式。默认情况下，包含 `assert(expression)` 形式的行会导致：如果 `expression` 计算为 `false`，程序将终止并显示诊断消息。我们可以用这个简单的例子来演示：

```cpp
// ExA_04.cpp
// 演示断言
import <iostream>;
import <format>;
#include <cassert>

int main() {
  int y {5};

  for (int x {}; x < 20; ++x) {
    std::cout << std::format("x = {}\ty = {}", x, y) << std::endl;
    assert(x < y);
  }
}
```

当 `x` 的值达到5时，你应该在输出中看到一条断言消息。当断言被触发时——在我们的例子中当 `x < y` 计算为 `false` 时——程序被终止。具体来说，`assert()` 宏调用标准库中的 `std::abort()` 函数，该函数立即终止程序。在终止之前，`assert()` 会在标准错误流 `cerr` 上输出一条诊断消息。该消息通常包含失败的条件以及失败发生的文件名和行号。

断言通常用于程序中的关键条件，如果某些条件不满足，灾难肯定会发生。你会希望确保如果出现这样的错误，程序不会继续运行。你可以使用任何逻辑表达式作为 `assert()` 宏的参数，因此你有很大的灵活性。

使用 `assert()` 简单而有效，当事情出错时，它提供了足够的信息来定位程序终止的位置。

> **警告**：`assert()` 用于检测编程错误，而不是用于处理运行时错误。它用于断言你作为程序员`110%`确信永远不应该发生的事实。因此，断言的逻辑表达式绝不应该基于你无法控制的事情。你的程序应包含代码来处理所有可能偶尔发生的错误情况。如第16章所述，异常是在C++中信号和/或处理此类运行时错误的有效手段。

### 第20页

**关闭 `assert()` 宏**

你可以通过在程序文件开头定义 `NDEBUG` 来在重新编译程序时关闭预处理器的断言机制：

```cpp
#define NDEBUG
```

这会导致翻译单元中的所有断言都被忽略。如果你在 ExA_04.cpp 的开头添加这个 `#define`，你将得到 `x` 从0到19的所有值的输出，并且没有诊断消息。请注意，只有在 `<cassert>` 的 `#include` 指令之前放置此指令才有效。

> **提示**：大多数编译器还允许你为所有源文件和头文件全局定义诸如 `NDEBUG` 之类的宏。通常，`NDEBUG` 是为完全优化的所谓“发布”配置定义的，但不为调试期间使用的配置定义。更多详情请查阅编译器文档。

> **警告**：如果定义了 `NDEBUG`，传递给 `assert()` 的逻辑表达式将不会被求值。因为它随后被预处理器移除，所以该表达式甚至不再是已编译代码的一部分。由于断言通常为优化编译而关闭，因此逻辑表达式的求值绝不应该引起任何副作用。

**静态断言**

与 `assert()` 宏不同，静态断言是C++语言的一部分。也就是说，它们不是标准库的补充，而是内置于语言中的。`assert()` 宏用于在运行时动态检查条件，而静态断言用于在编译时静态检查条件。通常，静态断言是以下两种形式之一的语句：

```cpp
static_assert(constant_expression);
static_assert(constant_expression, error_message);
```

`static_assert` 是一个关键字，`constant_expression` 必须在编译时产生一个可以转换为 `bool` 类型的结果，`error_message` 是一个可选的字符串字面量。如果 `constant_expression` 计算为 `false`，则程序的编译应失败。编译器将中止编译并输出一条诊断消息。当 `constant_expression` 为 `true` 时，静态断言什么也不做。

编译器需要能够在编译期间计算 `constant_expression`。这限制了你可以使用表达式的范围。典型的 `static_assert()` 表达式由字面量、由字面量初始化的 `const` 变量、宏、`sizeof()` 运算符、模板参数等组成。

### 第21页

> **注意**：随着每个新版本的C++标准，编译器在编译时能够计算的表达式和函数的范围正在增加。你可以通过简单地将 `constexpr` 关键字添加到它们的声明中，来定义各种函数、变量甚至 lambda 表达式，它们应该能够在编译时静态求值。

作为一个简短的例子，假设你的程序不支持32位编译，例如因为它需要寻址超过2GB的内存来处理更大的数据集。那么你可以在源文件中的任何位置放置以下静态断言：

```cpp
static_assert(sizeof(int*) > 4, "32-bit compilation is not supported.");
```

添加这个静态断言将确保你不会无意中编译成32位程序。

> **注意**：在C++20之前，静态断言常被用来限制模板的类型参数，通常是为了在这些模板不小心用不支持的类型参数实例化时产生更易读的错误消息。以下是一个例子：
> ```cpp
> template<typename T>
> T average(const std::vector<T>& values) {
>   static_assert(std::is_arithmetic_v<T>, "Type parameter for average() must be arithmetic.");
>   // ... 对 T 类型值应用运算符 + 和 / 的代码
> }
> ```
> 然而，从第21章你当然知道，类型约束和 `requires` 子句在约束模板类型参数方面更强大、更有效。以下是你如何在C++20中编写前面的例子：
> ```cpp
> template<typename T>
> concept Arithmetic = std::integral<T> || std::floating_point<T>;
>
> template <Arithmetic T>
> T average(const std::vector<T>& values) {
>   // ... 对 T 类型值应用运算符 + 和 / 的代码
> }
> ```

### 第22页

**包含文件**

我们要介绍的最后一个预处理指令是 `#include`。`#include` 指令获取给定文件的内容并将其注入当前文件。为了说明这一点，你可以创建一个名为 `inclusivity.quote` 的文件，其中包含以下三行代码：

```
std::cout << "We are trying to construct a more inclusive society.\n";
std::cout << "We are going to make a country in which no one is left out.\n"
          << "\t\t\t\t\t\t\t\t\t\t\t\t\t\t\t\t\t\t\t\t\tFranklin D. Roosevelt" << std::endl;
```

然后你可以通过 `#include` 指令将这些三行注入到一个实际的程序中，如下所示：

```cpp
// ExA_05.cpp
// 介绍 #include 指令
import <iostream>;

int main() {
#include "inclusivity.quote"
}
```

在预处理器交给编译器的翻译单元中，`#include` 指令将消失。它被 `inclusivity.quote` 文件的内容所替代。执行时，ExA_05 输出以下鼓舞人心的、至今仍非常切题的关于包容性的引言。

当然，我们在 ExA_05 中使用 `#include` 的方式并不是你通常使用该指令的方式。你通常使用 `#include` 来包含头文件。你也通常将所有 `#include` 指令放在文件顶部附近，而不是函数体中间。但 ExA_05 很好地说明了 `#include` 指令是什么：一个原始的、低级的构造，它指示预处理器查找具有给定名称的文件，获取其内容，并将其复制粘贴到翻译单元中找到 `#include` 指令的确切位置。仅此而已。正如我们稍后将看到的，这对翻译过程的其余部分，以及你应该如何编写和构建头文件有重大影响。但首先，关于 `#include` 指令的语法有一些额外的说明。

**双引号与尖括号**

以下是你如何将你自己名为 `inclusivity.quote` 的源文件包含到 ExA_05.cpp 的翻译单元中的方法：

```cpp
#include "inclusivity.quote"
```

这种语法与你之前用来包含标准库头文件的语法略有不同，后者看起来像这样：

```cpp
#include <cmath>
```

### 第23页

在双引号之间或尖括号之间指定文件名的具体区别在于预处理器将搜索具有该名称的文件的位置。通常，你的想法是使用尖括号来包含外部库的头文件，使用双引号来包含存储在当前源文件旁边的文件。

```cpp
#include <some_external_header.h>
#include "some_local_header.h"
```

通常，当文件名在双引号之间时，预处理器首先搜索当前目录。最终，如果在当前目录找不到该文件，预处理器需要回退到搜索如果文件名在尖括号之间时它会搜索的相同目录。

当一个名字在尖括号之间时，预处理器默认通常只搜索它知道包含标准库头文件的目录。大多数编译器也允许你提供其他目录供预处理器搜索。这些所谓的**附加包含目录**然后包含你的程序可能依赖的任何外部（通常是第三方）库的头文件。

> **提示**：大多数第三方库通过位于名为 `include` 的目录中的头文件使其 API 可用。

如果一个头文件位于某个子目录中，通常你会将头文件的相对路径放在引号或括号之间。例如：

```cpp
#include "subdirectory/in/current_directory/myheader.h"
#include <subdirectory/in/external_library_include_directory/some_header.h>
```

**理解头文件包含**

在本附录的后面，我们将向你展示如何正确编写你自己的头文件。在此之前，你首先需要理解其局限性以及一些更底层的概念。尽管如此，我们相信在此先让你直观感受头文件包含是如何工作的，这将使你更容易理解我们接下来需要解释的、更具理论性的章节。

考虑这个熟悉的 `#include` 指令，你已经多次使用它来访问标准库的基本数学函数：

```cpp
#include <cmath>
```

在第一步中，预处理器用 `<cmath>` 标准库头文件的内容替换这个 `#include` 指令。这个头文件的确切位置是实现定义的。

与之前的宏文本替换一样，任何被包含的文本体也会立即被预处理，这通常会导致更多文件的内容被注入翻译单元。依此类推，直到没有预处理指令剩下。

### 第24页

即使 `<cmath>` 可以说是较为简单的标准库头文件之一，一个 `<cmath>` 头文件通常也包含多个其他头文件的 `#include` 指令，而这些头文件又依赖于更多的头文件，依此类推。在我们查看的标准库实现中，仅仅一个 `<cmath>` 的 `#include` 指令，完全展开后，会导致大约6000行代码被注入到我们的翻译单元中。考虑到你通常只想调用一两个简单的数学函数，这是相当庞大的。

这立即说明了 `#include` 指令的主要缺点之一：它们为预处理器、编译器和链接器产生了大量的工作，所有这些都必须一遍又一遍地处理无数的C++实体，其中大部分通常甚至不被你的应用程序使用。好消息是，尽管这显然会给编译时间带来压力，但 `#include` 对最终可执行文件大小的影响通常是有限的。在大多数情况下，你不使用的实体的代码不会进入最终的二进制文件。

这应该让你对头文件包含的工作原理有了足够的感受。简而言之，即使只包含一个头文件，也常常会将数千行代码注入到一个翻译单元中，其中包含数百个实体声明和定义。关于你因此被允许在你自己的头文件中放置哪些声明和定义，以及你应该如何稳健地做到这一点，只有在你了解了更多关于单一定义规则和不同类型的链接之后，我们才能正确解释。

**单一定义规则**

单一定义规则是C++中的一个重要概念。尽管它的名字如此，ODR 实际上并不是一个单一的规则；它更像是一组规则。我们对这些规则的阐述不会详尽无遗。我们的主要目的是让你熟悉 ODR 限制背后的一般思想。这将帮助你更好地理解如何将程序代码组织到头文件和源文件中，并解读和解决你在违反 ODR 规则时会遇到的编译器和链接器错误。

**翻译单元内的 ODR**

在给定的翻译单元中，任何变量、函数、类类型、枚举类型或模板都绝不能定义超过一次。你可以有多个任何此类实体的声明，但绝不能有多个定义来确定它是什么并导致其被创建。如果在同一个翻译单元内有多个定义，代码将无法编译。

> **注意**：声明将一个名字引入一个作用域。定义不仅引入名字，还定义它是什么。换句话说，所有定义都是声明，但并非所有声明都是定义。

### 第25页

为了熟悉 ODR 违规产生的错误消息，你可以尝试编译以下程序。

```cpp
// ExA_06.cpp
// 定义同一个函数两次
import <iostream>;
import <format>;

double power(double x, int n);
double power(double x, int n); // 冗余声明（无害）

int main() {
  for (int i {-3}; i <= 3; ++i)
    std::cout << std::format("{:10}\n", power(8.0, i));
  std::cout << std::endl;
}

double power(double x, int n) // 第一个定义
{
  if (n == 0) return 1.0;
  else if (n > 0) return x * power(x, n - 1);
  else /* n < 0 */ return 1.0 / power(x, -n);
}

double power(double x, int n); // 另一个冗余声明（无害）

double power(double x, int n) // 第二个、更高效的定义（错误！）
{
  if (n < 0) return 1.0 / power(x, -n);
  if (n == 0) return 1.0;
  const double y{ power(x, n / 2) };
  return y * y * (n % 2 == 1 ? x : 1.0);
}
```

ExA_06.cpp 包含同一个函数 `power()` 的多个声明和定义。尽管在同一个翻译单元中多次声明同一个函数是奇怪的，但C++完全允许这样做。你可以在一个翻译单元内拥有任意多个相同函数的声明，只要它们都是等价的，并且这些声明中不超过一个是定义。

显然，ExA_06.cpp 的问题在于它的两个 `power(double, int)` 定义。编译器如何知道它应该在目标文件中为哪个定义生成机器码？是顶部的朴素实现？还是底部更高效的实现？因此，ExA_06.cpp 的编译失败并报错——这是理所当然的！

当然，只有真正的糊涂虫才会在同一个文件中显式定义同一个实体两次。但是，重复定义有时确实会通过（直接或间接）包含一个创建不当的头文件而进入同一个翻译单元。我们将在本附录后面讨论你应该使用哪些技术来防止这种情况。

### 第26页

**程序范围内的 ODR**

ODR 规则不仅适用于一个翻译单元内。它们还规定，常规函数和变量在整个程序中必须且只能定义一次。通常，不允许对同一函数或变量有两个定义，即使它们是相同的并且出现在不同的翻译单元中。自然，违反 ODR 规则这一方面的情况只有在翻译的链接阶段才会被发现。

此规则的例外是**内联函数**和**内联变量**。事实上，对于未在模块文件中定义的内联函数和内联变量，在其使用的每个翻译单元中，必须且只能出现一次定义。不过，程序中不同翻译单元内的给定内联函数或变量的所有定义必须相同。传统上，人们使用内联函数和变量来定义头文件中的变量或函数，而对于常规函数和变量，这样做几乎肯定会导致 ODR 违规。我们稍后在准备讨论头文件时，会向你展示如何定义内联实体。

类类型、枚举类型和模板的程序范围 ODR 规则与内联函数和变量的规则类似。也就是说，除非它们在模块文件中定义，否则对于使用这些实体的每个翻译单元，都需要且只需要一个定义。请注意，在某种程度上，编译器通过为任何给定的模板实参组合仅实例化每个模板一次，来为模板保留类似 ODR 的行为。

因此，虽然允许多个相同的类类型定义，但其成员函数和变量必须遵守与常规函数和变量相同的 ODR 规则。换句话说，非内联类成员在整个程序中只能有一个定义。这就是为什么在非模块化代码中，你通常将类成员的定义放在源文件中，而将类定义本身放在相应的头文件中。我们稍后会看到一个例子。

**ODR 与模块**

在完全模块化的程序中，你遇到 ODR 问题的可能性要小得多。事实上，在模块化代码中，你遇到 ODR 违规的唯一方式是当多个模块导出同一个实体时——你应该能够通过将名字适当地组织在命名空间中来避免这种情况。

首先，ODR 违规在模块中不太可能发生的一个明显原因是，`import` 声明与 `#include` 指令有根本的不同。你放在头文件中的任何定义都会被复制粘贴到每个包含它的翻译单元中。因为这本质上会多次复制声明，所以在非模块化环境中你总是必须警惕 ODR 违规。然而，模块接口文件的内容永远不会被复制。`import` 声明是 C++ 语言的一部分，因此不需要由预处理器处理和移除。它只是告诉 C++ 编译器在消费翻译单元中使给定模块的声明可见和/或可访问；导入模块时永远不会复制任何定义。因此，模块化代码中的任何重复定义都将是你亲手造成的。

其次，你不太可能在模块化代码中遇到 ODR 问题的另一个原因是，命名模块 A 中的非导出实体总是与模块 B 中的非导出实体不同，即使这些实体看起来具有完全相同的名称或签名。形式上，模块局部实体被称为具有**模块链接**，而非模块化源文件中的局部实体通常默认具有**外部链接**。我们将在下一节进一步讨论这些和其他类型的链接。

### 第27页

**不同类型的链接**

一个翻译单元中定义的实体通常需要从另一个翻译单元中的代码访问，即使在非模块化代码中也是如此。函数和类类型是明显的例子，但你也可以有其他例子——例如，在全局作用域定义的、在多个翻译单元间共享的变量，或者类型别名或枚举类型的定义。因为编译器一次只处理一个翻译单元，所以编译器无法解析此类引用。只有当程序的所有翻译单元的目标文件都可用时，链接器才能做到这一点。

翻译单元中的名字在编译/链接过程中被处理的方式由一种称为**链接**的属性决定。一个名字的链接表达了它所代表的实体在程序代码中可以在哪里被定义和/或访问。你在程序中使用的每个名字要么有链接，要么没有。当你可以使用一个名字来访问程序中声明该名字的作用域之外的东西时，该名字就有链接。如果不是这种情况，它就没有链接。如果一个名字确实有链接，那么它可以具有**内部链接**、**模块链接**或**外部链接**。

**确定名字的链接**

翻译单元中每个名字的链接是在任何头文件的内容被插入到作为该翻译单元基础的源文件之后确定的。链接的可能性具有以下含义：

-   **内部链接**：该名字代表的实体只能从同一翻译单元内部访问。例如，在全局或命名空间作用域定义的、非内联、非导出、`const` 限定的变量的名字默认具有内部链接。
-   **外部链接**：具有外部链接的名字除了可以在其定义的翻译单元中访问外，还可以从其他翻译单元访问。换句话说，该名字代表的实体可以在整个程序中共享和访问。在非模块化代码中，大多数在全局或命名空间作用域声明的名字默认具有外部链接——包括函数、枚举、类、模板以及内联或非 `const` 变量的名字。当然，所有从模块导出的实体也具有外部链接。
-   **模块链接**：具有模块链接的实体名字只能在同一个命名模块的翻译单元中访问。在模块文件的管辖范围内（即，在模块文件的可选全局模块片段之后的任何地方）的全局或命名空间作用域声明的任何非导出实体的名字默认具有模块链接。
-   **无链接**：当一个名字没有链接时，它所指代的实体只能从适用于该名字的作用域内访问。所有在块内定义的名字——即局部名字——都没有链接。

**外部函数**

在由多个文件组成的程序中，链接器建立（或解析）一个源文件中的函数调用与另一个源文件中的函数定义之间的连接。当编译器编译对函数的调用时，它只需要函数原型中包含的信息来创建调用。编译器要么从（预编译的）模块接口文件中提取此原型，要么从当前翻译单元本身存在的任何函数声明中提取（通常是在从头文件包含之后）。

### 第28页

无论哪种方式，编译器都不关心函数的定义是出现在同一个翻译单元中，还是出现在任何其他翻译单元中。如果一个函数没有在调用它的翻译单元中定义，编译器会将该调用标记为外部，并留给链接器去解决。

我们应该用一个例子来阐明这一点。为此，我们将改编 ExA_01.cpp，并将其 `power()` 函数的一个定义移动到一个不同的翻译单元。`power()` 的另一个定义我们直接丢弃。

```cpp
// ExA_07.cpp
// 调用外部函数
import <iostream>;
import <format>;

double power(double x, int n); // 声明一个外部 power() 函数

int main() {
  for (int i {-3}; i <= 3; ++i)
    std::cout << std::format("{:10}\n", power(8.0, i));
  std::cout << std::endl;
}
```

尽管 `main()` 调用了 `power()`，但 ExA_07 翻译单元中不存在该函数的定义。但这没关系。如前所述，编译器执行对 `power()` 的调用所需的一切就是它的原型。因为函数名默认具有外部链接，编译器只是在 ExA_07 翻译单元的目标文件内部记录了一个对外部定义的 `power()` 函数的调用。然后，链接器的任务就是将这个调用与一个定义连接起来。如果链接器在程序的其他翻译单元之一中没有找到合适的定义，它将将此标记为翻译失败。

为了使 ExA_07 程序正确翻译，你需要第二个包含 `power()` 定义的翻译单元。我们当然选择更高效的实现。

```cpp
// Power.cpp
// 在 ExA_07.cpp 中调用的 power 函数在另一个不同的翻译单元中定义
double power(double x, int n) {
  if (n < 0) return 1.0 / power(x, -n);
  if (n == 0) return 1.0;
  const double y{ power(x, n / 2) };
  return y * y * (n % 2 == 1 ? x : 1.0);
}
```

通过链接 ExA_07 和 Power 翻译单元的目标文件，你获得了一个与 ExA_01 的程序完全等效的程序。

请注意，为了使用 `power()` 函数，我们仍然需要在 ExA_07.cpp 的开头向编译器提供一个原型。如果你必须为每个外部定义的函数显式地这样做，并且此外你必须在每个想要调用它们的源文件中重复这些相同的原型，那将是非常不切实际的。这就是为什么在非模块化代码中，函数原型通常被收集在头文件中，然后你可以方便地将它们 `#include` 到你的翻译单元中。以下是对其工作原理的预览。

### 第29页

```cpp
// Power.h
// 你的第一个头文件，包含一个外部 power() 函数的声明
double power(double x, int n);
```

```cpp
// ExA_07A.cpp
// 调用在头文件中声明的外部函数
import <iostream>;
import <format>;
#include "Power.h"

int main() {
  for (int i {-3}; i <= 3; ++i)
    std::cout << std::format("{:10}\n", power(8.0, i));
  std::cout << std::endl;
}
```

ExA_07A 和 ExA_07 之间唯一的区别是，我们将 `power()` 的声明移到了一个头文件 Power.h 中，随后我们在 ExA_07A.cpp 中包含该头文件。通过包含 Power.h，预处理器显然为 ExA_07A.cpp 产生了与 ExA_07.cpp 完全相同的翻译单元，因此编译器甚至没有注意到两个程序之间的区别。然而，作为开发者的你，现在只需要输入 `power()` 的原型两次——一次在头文件中声明，一次在源文件中定义——这显然比在你想要调用该函数的每个源文件中输入一次前进了一步。当然，如果你需要在每个使用该类的翻译单元中获得整个类定义的副本，这种收益会更大，但我们稍后会讲到。

这个朴素的头文件 Power.h 是完美正确和安全的，只因为它不包含任何定义。如果你的头文件确实包含定义（类定义、模板定义、变量或函数的内联定义等），事情就会变得稍微复杂一些。如果没有额外的预防措施，头文件包含将很快导致 ODR 违规，即相同的定义被无意中多次包含在同一个单元中。在本附录后面，我们将解释如何避免此类问题。

在我们向你展示如何创建更高级的头文件之前，我们首先继续讨论各种类型的链接，现在从外部函数转向外部变量。

**外部变量**

假设在 ExA_07.cpp 中，你想使用一个外部定义的变量 `power_range` 来替换魔法常量 `-3` 和 `3`，像这样：

```cpp
for (int i {-power_range}; i <= power_range; ++i)
  std::cout << std::format("{:10}\n", power(8.0, i));
```

第一步是创建一个名为 `Range.cpp` 的额外源文件，其中包含变量的定义：

```cpp
// Range.cpp
int power_range{ 3 }; // 一个具有外部链接的全局变量
```

### 第30页

非 `const` 变量默认具有外部链接，就像函数一样。因此，其他翻译单元访问该变量不会有问题。然而，有趣的问题是：如何在消费翻译单元中声明一个变量而不使其成为第二个定义？一个合理的第一次尝试是这样的：

```cpp
// ExA_08.cpp
// 使用一个外部定义的变量
import <iostream>;
import <format>;

double power(double x, int n); // 声明一个外部 power() 函数
int power_range;               // 一个不算不合理的第一次尝试，对吧？

int main() {
  for (int i {-power_range}; i <= power_range; ++i)
    std::cout << std::format("{:10}\n", power(8.0, i));
  std::cout << std::endl;
}
```

编译器对 `power_range` 的声明不会有问题。然而，链接器会报错！我们建议你也尝试一下，以熟悉这个错误消息。原则上，你应该能够推断出我们为 `power_range` 提供了两个不同的定义：一个在 Range.cpp 中，一个在 ExA_08.cpp 中。这，当然，违反了单一定义规则！

潜在的问题是，我们在 ExA_08.cpp 中对 `power_range` 变量的声明不仅仅是任何旧的变量声明；它是一个**变量定义**：

```cpp
int power_range;
```

实际上，你可能已经知道会发生这种情况。当然，你会记得，如果你忽略初始化变量，变量通常包含垃圾。然而，在第3章末尾，我们也介绍了全局变量。正如我们告诉你的那样，全局变量将被初始化为零，即使你省略了定义中的花括号初始化器。换句话说，我们在 ExA_08.cpp 中对全局变量 `power_range` 的声明等同于以下定义：

```cpp
int power_range {};
```

而 ODR 不允许同一个变量有两个定义。因此，编译器需要被告知，全局变量 `power_range` 的定义将在当前翻译单元 ExA_08 的外部。如果你想访问一个在当前翻译单元外部定义的变量，那么你必须使用 `extern` 关键字声明该变量名：

```cpp
extern int power_range; // 一个外部定义的变量的声明
```

这个语句是一个声明，表明 `power_range` 是一个在别处定义的名字。类型必须与定义中出现的类型完全对应。你不能在 `extern` 声明中指定初始值，因为它是名字的声明，而不是变量的定义。将变量声明为 `extern` 意味着它在另一个翻译单元中定义。这导致编译器标记外部定义的变量的使用。是链接器在名字和它所引用的变量之间建立连接。

### 第31页

在这个声明出现的任何块内，名字 `power_range` 引用在另一个文件中定义的变量。该声明可以出现在任何需要访问 `power_range` 的翻译单元中。你可以将声明放在全局作用域，以便在整个翻译单元中可用，或者放在一个块内，在这种情况下，它仅在局部作用域内可用。

> **注意**：你也可以在函数声明前添加 `extern` 说明符。例如，在 ExA_08.cpp 中，你可以使用显式的 `extern` 说明符声明 `power()` 函数，以提醒该函数的定义将是另一个翻译单元的一部分：
> ```cpp
> extern double power(double x, int n);
> extern int power_range;
> ```
> 虽然这完全是可选的，但我们的建议是，在任何外部定义的函数的声明前添加 `extern`，除非该声明出现在头文件或模块接口文件中。这样做不仅一致，而且对阅读代码的人来说也更清晰。

**具有外部链接的 `const` 变量**

考虑到其性质，你当然希望将 ExA_08 的 Range.cpp 中的 `power_range` 变量定义为全局常量，而不是可修改的全局变量：

```cpp
// Range.cpp
const int power_range {3};
```

然而，`const` 变量默认具有内部链接，这使得它在其他翻译单元中不可用。你可以通过在定义中使用 `extern` 关键字来覆盖这一点：

```cpp
// Range.cpp
extern const int power_range {3}; // 一个具有外部链接的全局常量的定义
```

`extern` 关键字告诉编译器，尽管是 `const`，该名字应该具有外部链接。当你想在另一个源文件中访问 `power_range` 时，你必须将其声明为 `const` 和 `external`：

```cpp
extern const int power_range; // 一个外部全局常量的声明
```

你可以在完全正常工作的示例 ExA_08A 中找到这一点。

全局变量对于你想共享的常量值可能很有用，因为它们在任何翻译单元中都是可访问的。通过在所有需要访问它们的程序文件之间共享常量值，你可以确保在你的程序中为这些常量使用相同的值。然而，与外部函数一样，如果多个非模块化文件中需要一个全局变量，你通常会将它们放在一个头文件中。你将在本附录后面看到一个例子。

> **提示**：正如第3章所述，编码和设计指南通常规定应避免使用全局变量。然而，全局常量——用 `const` 关键字声明的全局变量——是这条规则的一个可敬的例外。建议只定义一次你的所有常量（在头文件或模块文件中），而全局变量非常适合这一点。

### 第32页

**内部名字**

如果有办法指定名字应该具有外部链接，那么肯定也有办法指定它们应该具有内部链接，对吧？当然有。甚至有两种。但都不是你期望的那一种！

让我们首先说明何时以及为何需要这种可能性。也许你注意到在 ExA_07 中，每次递归调用 `power()` 时，该函数都会检查其参数 `n` 是正数还是负数。这有点浪费，因为 `n` 的符号当然永远不会改变。一个选项是以如下方式重写 `power()`：

```cpp
// Power.cpp
// 一个稍微优化的 power 函数

double localHelper(double x, unsigned n) {
  if (n == 0) return 1.0;
  const double y{ localHelper(x, n / 2) };
  return y * y * (n % 2 == 1 ? x : 1.0);
}

double power(double x, int n) {
  return n >= 0 ? localHelper(x, static_cast<unsigned>(n)) : 1.0 / localHelper(x, static_cast<unsigned>(-n));
}
```

`power()` 函数本身现在不再是递归的。相反，它调用递归辅助函数 `localHelper()`，该函数被定义为仅对正指数有效。

在这种情况下，`localHelper()` 原则上可以是一个有用的函数本身——最好重命名，以成为针对 `unsigned` 指数的 `power()` 的重载。然而，为了便于讨论，假设我们希望 `localHelper()` 仅仅是一个局部辅助函数，一个只能被 `power()` 调用的函数。你会发现这种需求经常出现；你需要一个函数来使你的局部代码更清晰或在某个特定的翻译单元中重用，但该函数太特殊，无法导出到程序的其余部分以供重用。

我们当前的 `localHelper()` 函数和 `power()` 一样，也具有外部链接，因此可以从任何翻译单元内部调用。更糟糕的是，单一定义规则意味着其他翻译单元也不能再定义具有相同签名的 `localHelper()` 函数了！如果所有的局部辅助函数总是具有外部链接，那么它们也需要唯一的名字，这使得在大型程序中避免名字冲突变得非常困难。

我们需要一种方法告诉编译器，像 `localHelper()` 这样的函数应该具有内部链接而不是外部链接。鉴于 `extern` 赋予名字外部链接，一个明显的尝试是在定义前添加一个 `intern` 说明符。如果C++中没有这样的关键字，这可能就奏效了。相反，在（非常）早期的日子里，标记一个名字（函数名或变量名）具有内部链接的方法是添加 `static` 关键字。以下是一个例子：

```cpp
static double localHelper(double x, unsigned n) { // localHelper() 现在具有内部链接
  if (n == 0) return 1.0;
  const double y{ localHelper(x, n / 2) };
  return y * y * (n % 2 == 1 ? x : 1.0);
}
```

### 第33页

虽然这种表示法仍然有效，但不再推荐使用 `static` 关键字。这种源自C语言的语法之所以没有被C++标准弃用或移除，唯一的原因是你仍然会在遗留代码中大量看到它。然而，在C++中，定义具有内部链接的名字的推荐方法是通过**无名命名空间**。

> **警告**：不要再使用 `static` 来标记应该具有内部链接的名字；始终使用无名命名空间代替。

**无名命名空间**

你可以使用以下代码声明一个无名命名空间：

```cpp
namespace {
  // 命名空间中的代码、函数等
}
```

在无名命名空间内声明一个实体，其效果等同于在全局命名空间中将其声明为 `static`，除了 `static` 只能应用于变量或函数。在一个翻译单元内，你可以像引用在全局命名空间中定义的名字一样引用其无名命名空间的实体。以下是你通常会在我们正在进行的例子的 Power.cpp 源文件中创建局部辅助函数的方法：

```cpp
// Power.cpp
// 一个具有内部链接的局部辅助函数
namespace {
  double localHelper(double x, unsigned n) { // localHelper() 具有内部链接
    if (n == 0) return 1.0;
    const double y{ localHelper(x, n / 2) };
    return y * y * (n % 2 == 1 ? x : 1.0);
  }
}

double power(double x, int n) { // power() 具有外部链接
  return n >= 0 ? localHelper(x, static_cast<unsigned>(n)) : 1.0 / localHelper(x, static_cast<unsigned>(-n));
}
```

### 第34页

具有内部链接的实体永远不能从翻译单元外部访问。类似于 `localHelper()` 和 `power()` 中的参数名 `x` 引用两个不同的实体，任何在 Power 以外的翻译单元中具有原型 `double localHelper(double, unsigned)` 的函数声明现在将不再命名同一个实体。你可以通过如下修改 ExA_07 的主源文件来尝试：

```cpp
// ExA_09A.cpp
// 尝试从不同的翻译单元调用一个具有内部链接的函数
import <iostream>;
import <format>;

double localHelper(double x, unsigned n); // 声明一个外部 localHelper() 函数

int main() {
  for (unsigned i {0}; i <= 5; ++i)
    std::cout << std::format("{:10}\n", localHelper(8.0, i));
  std::cout << std::endl;
}
```

ExA_09A 中的声明现在声明了一个外部函数 `localHelper()`——链接器将找不到其定义的函数。

> **提示**：在无名命名空间内部声明的所有名字都具有内部链接。如果一个函数、全局变量、类、枚举或模板不应该从特定翻译单元外部访问，你应该始终在无名命名空间中定义它，尤其是在非模块化代码中。为此目的使用 `static` 说明符已不再推荐。

**模块局部名字**

每个模块在某种程度上都形成了自己自包含的小宇宙，在其中 ODR 适用。自然，导出实体的名字具有外部链接，这意味着两个模块导出同一个实体构成 ODR 违规。但是，在模块文件的管辖范围内的全局或命名空间作用域声明的非导出实体的名字默认具有**模块链接**，而不是外部链接。这意味着，如果多个模块各自定义了一个具有相同原型的模块局部函数，链接器不会将它们视为同一个实体，因此不会导致违反任何 ODR 规则。以下示例说明了模块链接的含义：

```cpp
// ModuleA.cpp
export module A;

void logError(std::string_view error) { /* ... */ } // 模块局部函数
```

```cpp
// ModuleB.cpp
export module B;

void logError(std::string_view error) { /* ... */ } // 模块局部函数
```

### 第35页

```cpp
// ModuleCSource.cpp
module C;

void logError(std::string_view error) { /* ... */ } // 模块局部函数
```

如果你只在某个模块翻译单元中需要一个特定的实体，你当然仍然可以通过在无名命名空间中声明它来赋予其内部链接。尽管如此，在模块化代码中使用无名命名空间来避免 ODR 违规不那么关键。毕竟，在同一个模块内无意中定义两个相同的实体（模块通常要小得多，并且完全由同一个开发者或开发团队控制）比在整个程序内遇到 ODR 违规的可能性要小得多。

**组织非模块化代码**

既然你已经理解了预处理指令、ODR 规则和不同类型的链接，我们终于准备好解释如何正确地将非模块化代码组织到头文件和源文件中。

> **注意**：如果你的编译器支持，我们强烈建议你始终在新代码中使用模块，并尽可能地将现有代码转换为模块。但这并不总是一个选项，即使它是，你很可能仍然需要在遗留的非模块化代码中工作很长一段时间。所以，无论哪种方式，理解头文件和源文件如何相互关联无疑仍然是重要的。

我们已经简要介绍了头文件包含在 ExA_07A 中的工作方式，我们在其中包含了以下基本头文件，以避免在你想调用这个外部函数的每个源文件中重复 `power()` 的原型。

```cpp
// Power.h
// 你的第一个头文件，包含一个外部 power() 函数的声明
double power(double x, int n);
```

包含一个不包含任何定义的头文件（例如 Power.h）永远不会导致单一定义规则的问题。没有重复的定义，就没有 ODR 违规——这一点很清楚。接下来，我们来看看如果你以幼稚的方式向头文件添加定义时会发生的两种类型的 ODR 违规，当然我们会教你如何避免这些问题。

### 第36页

**防止头文件内容重复**

对于你的第一个非平凡头文件，我们从你创建的第一个模块文件开始是很合适的：`math.cppm`。以下是你如何将该模块接口文件转换为头文件的方法。

```cpp
// BadMath.h - 你第一个真正的头文件的开始
auto square(const auto& x) { return x * x; } // 简写函数模板

const double lambda{1.303577269034296391257 }; // Conway常数

enum class Oddity { Even, Odd };
bool isOdd(int x) { return x % 2 != 0; }
auto getOddity(int x) { return isOdd(x) ? Oddity::Odd : Oddity::Even; }
```

然而，如果你将这个头文件两次包含到同一个翻译单元中，你当然会违反最明显的 ODR 规则：在同一个翻译单元中，永远不允许同一个实体有两个定义。事实上，预处理之后，你基本上会陷入与 ExA_06 完全相同的情况。

```cpp
// ExA_10.cpp
// 包含相同的定义两次
import <iostream>;
#include "BadMath.h"
#include "BadMath.h"

int main() {
  std::cout << square(1.234) << std::endl;
}
```

当然，只有真正的糊涂虫才会像这样显式地将同一个头文件两次包含到同一个源文件中。但是，如前所述，你包含的每个头文件通常都会包含其他几个头文件，这个过程可以深入很多层。因此，即使在最基本的程序中，你也很有可能间接地在同一个翻译单元中包含头文件不止一次。在某些情况下，这甚至可能是无法避免的。

因此，向头文件添加定义显然为通过间接头文件包含导致 ODR 违规敞开了大门。我们的挑战是：我们如何再次关闭那扇门？

首先，一个显而易见的想法是简单地将所有定义移动到一个源文件中，只在头文件中留下非定义的声明。但是，不幸的是，这个想法只适用于函数和变量；它不适用于类、枚举类型、类型别名、模板等等。对于所有这些其他实体，定义——一个且只有一个——需要在每个使用这些实体的翻译单元中存在。由于这个原因，你不能将 `square()`（一个模板）或 `Oddity`（一个类型定义）的定义移出 ExA_10 的 BadMath.h 头文件并放入某个源文件。这完全类似于为什么在 Ex11_01 中你不能将这些相同的定义移出 `math` 模块的接口并移入模块实现文件。

### 第37页

那么，到目前为止我们得到了什么？一方面，每个头文件都不可避免地最终会在某个翻译单元中被包含不止一次。这与基本的 ODR 规则相结合，表明你永远不应该在头文件中定义任何实体。然而，另一方面，当你定义一个类型或模板时，你当然常常希望在一个以上的翻译单元中使用它。虽然一遍又一遍地定义 `Box` 类一直很有趣，但你通常希望只定义一次给定的类，然后在你的整个程序中重用该定义。在非模块化 C++ 中，这意味着你别无选择，只能在头文件中定义它，然后在需要的地方包含它。

在标准的、惯用的 C++ 中，这个看似僵局的解决方案是给每个头文件加上所谓的 **`#include` 守卫**。这种模式基于对 `#ifndef` 和 `#define` 预处理指令的巧妙使用。你已经看到，在定义宏时不必指定一个值：

```cpp
#define MY_IDENTIFIER
```

这会创建 `MY_IDENTIFIER`，因此它从此存在，并表示一个空字符序列。你也知道你可以使用 `#ifndef` 指令来测试给定的标识符是否已被定义，并根据结果在文件中包含或不包含代码。

`#ifndef` 之后直到 `#endif` 指令的所有行将完全从文件中省略，如果标识符 `MY_IDENTIFIER` 之前已被定义。它将不再被预处理器考虑，因此当然也不会被编译器考虑。如果 `MY_IDENTIFIER` 被定义，就好像那个条件代码块中的任何潜在的重复实体定义从未存在过一样。

这些观察结果构成了 `#include` 守卫的基础，你可以使用它们来确保头文件的内容在翻译单元中永远不会被复制。应用于之前的 BadMath.h 头文件，这种模式如下所示：

```cpp
// BetterMath.h - 创建头文件的第二次、更好的尝试
#ifndef BETTER_MATH_H
#define BETTER_MATH_H

auto square(const auto& x) { return x * x; } // 简写函数模板

const double lambda{ 1.303577269034296391257 }; // Conway常数

enum class Oddity { Even, Odd };
bool isOdd(int x) { return x % 2 != 0; }
auto getOddity(int x) { return isOdd(x) ? Oddity::Odd : Oddity::Even; }

#endif // #include 守卫结束
```

当然，你可以选择任何唯一的标识符来代替 `BETTER_MATH_H`，只要你在每个头文件中使用不同的标识符。尽管大多数标识符基于头文件本身的名称，但使用了不同的命名约定。在本附录中，我们将使用 `HEADER_NAME_H` 形式的标识符。

### 第38页

为了显示 `#include` 守卫的效果，我们现在将 BetterMath.h 两次包含到同一个翻译单元中，并带你一步步了解发生了什么。

```cpp
// ExA_10A.cpp
// 在同一个翻译单元中包含同一个头文件两次，而不违反 ODR
import <iostream>;
#include "BetterMath.h"
#include "BetterMath.h"

int main() {
  std::cout << square(1.234) << std::endl;
}
```

第一个 `#include` 指令将包含 BetterMath.h 中的各种 C++ 实体定义，因为此时 `BETTER_MATH_H` 尚未定义。在此过程中，预处理器定义了宏 `BETTER_MATH_H`。因此，在同一翻译单元中，任何后续的 `#include` 指令将不再包含任何代码，因为 `BETTER_MATH_H` 之前已被定义。就好像第二个 `#include` 指令从未存在过。再看看 BetterMath.h 中的 `#include` 守卫，我们相信你会欣赏这个技巧的巧妙之处。

> **提示**：用 `#include` 守卫包围你所有的头文件，以消除当头文件在同一个翻译单元中被包含两次时违反单一定义规则的可能性。

> **提示**：大多数编译器提供了一个 `#pragma` 预处理指令，以实现与相对冗长的 `#include` 守卫模式相同的效果。简单地在头文件开头放置一行包含 `#pragma once` 通常也足以防止头文件内容的重复。虽然几乎所有编译器都支持这个 `#pragma`，但它不是标准的 C++，因此在本附录中，我们将在头文件中继续使用 `#include` 守卫。

**在头文件中定义函数和变量**

虽然 `#include` 守卫可以防止单个翻译单元内的重复定义，但当你将相同的变量或函数定义包含到同一程序的多个翻译单元中时，它们并不能防止违反程序范围的 ODR 规则。

ExA_11 的主要目的是说明：因为 Hypot.cpp 和 Pow4.cpp 都包含一个针对 BetterMath.h 的 `#include` 指令，所以它们的翻译单元都包含了 `isOdd()` 和 `getOddity()` 函数的定义。这违反了程序范围的 ODR 规则，该规则规定在整个程序中只允许一个相同函数的定义。并且由于 C++ 预处理器独立地考虑每个翻译单元，再多的 `#include` 守卫或 `#pragma` 也无法保护你免受此影响。

### 第39页

> **注意**：回想一下，程序范围的 ODR 规则只限制函数和非 `const` 变量的定义。例如，在 ExA_11 中，只有 `isOdd()` 和 `getOddity()` 的定义违反了 ODR。对于同一个类型或模板的多个定义（例如枚举类 `Oddity` 或 `square()` 模板），ODR 规则是允许的，只要它们完全相同并且每个翻译单元最多有一个这样的定义。`lambda` 变量在 ExA_11 中没有违反 ODR 的原因是它被限定为 `const`，这在包含它的两个翻译单元中都赋予了其名字内部链接。因此，得到的 `lambda` 定义不被视为同一个外部实体的两个定义，而是被视为两个不同常量的独立定义，每个都对其各自的翻译单元是内部的。

这是否意味着你永远不能在头文件中定义函数和非 `const` 全局变量？你是否应该总是在源文件中定义函数和变量？当然不是。我们已经在关于 ODR 的部分给出了答案：程序范围的 ODR 规则仅适用于常规函数和常规变量；它们不适用于**内联函数**和**内联变量**。对于内联实体，程序中允许多个定义，只要它们完全相同，并且同一个翻译单元内不出现两个定义——就像类型和类定义一样。

但是，在我们向你展示如何创建头文件中的内联定义之前，也许首先简要讨论一下，除了避免多次输入相同的声明之外，你首先会将某些定义放在头文件中的原因和时机——这本身已经是一个完全可以理解的想法。

**内联**

在头文件中定义函数可能会带来更好的运行时性能。粗略地说，函数调用涉及到相当多的工作。然而，如果编译器可以访问函数定义，它可以简单地将函数体中的代码复制粘贴到调用函数中，并避免所有这些额外的步骤。这种编译器优化称为**内联**。

> **提示**：对于大多数函数，函数调用的开销与执行函数体所需的时间相比要么可以忽略不计，要么它们从未被足够频繁地调用以使内联对程序的整体性能产生任何影响。因此，你应该只考虑为那些既小又可能在时间关键型代码中被多次调用的函数促进编译时内联。

### 第40页

在头文件中定义函数和变量是有代价的，尤其是在开发过程中。每次你更改头文件中的定义时，需要重新编译所有包含该头文件的翻译单元。另一方面，如果你更改源文件中的定义，只需要重新编译该源文件。由于函数定义的更改频率远高于其原型，在源文件中而不是头文件中定义函数通常能极大地加快开发速度。

模块不像头文件那样会受到这个问题的严重影响，因此考虑因素不同。我们将在下一节进一步讨论。

**模块中的内联函数和变量**

如你所知，模块被设计为自包含、独立、可组合的单元。设计模块语言特性的关键目标之一是模块应该减少编译时间。这些担忧导致了这样的限制：只要模块的接口不变，你就应该能够在该模块内工作并随意重新编译其文件，而无需重新编译任何导入它的外部翻译单元。只有当你更改模块的导出接口时，你才需要重建其二进制接口文件，然后重新编译所有使用该接口的翻译单元。

因此，默认情况下，你的编译器不允许内联从不同模块导入的函数的调用。默认情况下，函数的定义不是模块接口的一部分，即使该定义出现在模块接口文件中。只有当你明确将导出函数的定义标记为 `inline` 时，它才成为模块接口的一部分，并且只有到那时，编译器才被允许在使用翻译单元中内联此定义。

请注意，将函数的定义包含在模块接口中确实意味着对其函数体的任何后续更改都需要重新编译使用翻译单元，因此即使在模块化代码中，你也只应该考虑对很小、其定义不太可能更改且在时间关键型代码中可能被大量调用的函数使用 `inline`。

**在头文件中定义类**

到目前为止，在本附录中，我们主要关注函数和变量的定义，主要是因为它们很紧凑。当然，非模块化 C++ 代码至少同样频繁地包含类定义。在本节中，我们将简要探讨定义类及其成员的头文件和源文件通常的样子。

### 第41页

因为你需要在使用该类对象的每个翻译单元中访问该类的定义，所以类本身通常定义在头文件中。以下面的头文件为例，它包含一个长期受欢迎的 `Box` 类的定义。

```cpp
// Box.h
// 你的第一个包含类定义的头文件
#ifndef BOX_H
#define BOX_H

class Box {
public:
  Box();
  Box(double length, double width, double height);

  double getLength() const;
  double getWidth() const;
  double getHeight() const;

  double volume() const;

private:
  double m_length;
  double m_width;
  double m_height;
};

#endif
```

因为成员定义必须遵守与常规函数和变量相同的 ODR 规则，所以在这个头文件的第一个版本中，我们保持了安全和简单，将所有成员函数定义在一个源文件中。除了顶部的 `#include` 指令和缺少模块声明之外，这个源文件看起来与类似的模块实现文件完全一样。

```cpp
// Box.cpp
#include "Box.h"

Box::Box() : Box(1.0, 1.0, 1.0) {
}

Box::Box(double length, double width, double height)
  : m_length(length), m_width(width), m_height(height) {
}

double Box::getLength() const { return m_length; }
double Box::getWidth() const { return m_width; }
double Box::getHeight() const { return m_height; }

double Box::volume() const { return m_length * m_width * m_height; }
```

### 第42页

以下是一个小的测试程序，向你展示这种非模块化的接口和定义分离是可行的。

```cpp
// ExA_13.cpp - 从头文件包含一个类定义
import <iostream>;
#include "Box.h"

int main() {
  Box boxy{1,2,3};
  std::cout << boxy.volume() << std::endl;
}
```

请注意，这个 `main()` 函数再次能够调用 `volume()` 成员函数，即使其翻译单元中不存在该成员的定义。这之所以有效，原因当然是编译器再次将此调用标记为外部函数调用，之后链接器将其与 `Box` 翻译单元中的正确定义匹配。你可以从 Box.cpp 中删除 `volume()` 的定义，看看链接器会产生什么错误。

**内联类成员**

你也可以直接在头文件中定义类成员函数，但前提是使用内联定义。

```cpp
// Box.h
// 内联类成员定义
#ifndef BOX_H
#define BOX_H

#include <ostream> // 注意：不要在这里使用 import <ostream>;

class Box {
public:
  Box() = default; // 类内定义，因此隐式内联
  Box(double length, double width, double height);

  // 类内成员定义是隐式内联的
  double getLength() const { return m_length; };
  double getWidth() const { return m_width; };
  double getHeight() const { return m_height; };

  double volume() const;

private:
  double m_length {1.0};
  double m_width {1.0};
  double m_height {1.0};
};

// 类外成员定义必须显式标记为内联
inline Box::Box(double length, double width, double height)
  : m_length(length), m_width(width), m_height(height) {
}

inline double Box::volume() const {
  return m_length * m_width * m_height;
}

// 非成员函数的定义当然也必须是内联的
inline std::ostream& operator<<(std::ostream& out, const Box& box) {
  return out << "Box(" << box.getLength() << ", " << box.getWidth() << ", " << box.getHeight() << ')';
}

#endif // #include 守卫结束
```

在全局模块中，所有类内成员定义都是隐式内联的。这就是为什么在 Box.h 中的默认构造函数或三个 getter 前面不需要 `inline`。然而，在类外定义成员时，只有当你明确将其标记为 `inline` 时，其定义才是内联的。如果没有在非默认构造函数或 `Box::volume()` 的定义前加上 `inline` 说明符，ODR 将禁止你在同一程序的多个翻译单元中包含 Box.h。

### 第43页

请注意，我们还在 Box.h 中添加了一个流输出运算符的非成员运算符重载。因为这个定义在头文件内部，你当然也必须在那里添加 `inline` 说明符。还要注意，我们使用 `#include` 指令而不是 `import` 声明来在 Box.h 中获取 `std::ostream` 类的定义。实际上，你甚至不允许在这里使用 `import` 声明。原因如下：

> **警告**：永远不要在头文件中添加模块导入声明。根据设计，构建系统应该能够在不进行完全预处理的情况下发现模块依赖关系。这意味着 `import` 声明不允许源自 `#include` 指令或宏展开。这也与为什么所有 `import` 声明必须在模块的管辖范围内、在任何其他声明之前出现有关。所有这些限制都是为了便于高效的依赖关系发现而设立的，这是构建系统在决定编译大量程序文件的顺序时需要做的事情。

**模块与头文件**

我们已经在第11章或本附录中介绍了模块相对于头文件提供的大部分优势。我们通过简要回顾这些来结束这最后一节。

-   **模块导致编译时间减少**，因为每个模块文件只被编译器处理一次。而头文件在每个直接或间接使用它们的翻译单元中被反复包含和处理。
-   **模块也导致更快的增量重新编译**，因为只有对模块接口的更改才会触发对使用翻译单元的重新编译。对非内联函数体或局部实体的更改只需要重新编译模块文件本身。
-   **模块是自隔离的**，这意味着只有对模块接口的更改才能影响其他翻译单元的编译。头文件常常将局部实体的声明和定义，以及底层头文件的 `#include` 指令或有危险的宏定义泄漏到包含它们的翻译单元中——所有这些都可能导致意外的名称冲突或其他编译问题。想想 Windows API 头文件定义的 `min()` 和 `max()` 宏在 C++ 代码中经常造成的破坏！
-   **你导入模块的顺序从不重要**，而头文件则可能不同。原因是模块接口永远不会受到导入翻译单元中任何宏定义的影响，也永远不会注入任何新的宏定义。
-   使用模块，你可以**非常容易和自然地在单个文件中定义所有实体**。不需要诸如 `#include` 守卫、`#pragma`、无名命名空间或 `inline` 说明符等结构——至少不是为了避免 ODR 违规。
-   所有这些综合起来应该让一件事变得非常清楚：如果你的编译器和构建系统支持，你应该始终使用模块来组织新代码。毫无疑问。

### 第44页

**头文件单元**

在关于组织非模块化代码的部分，我们将一个模块接口文件转换为头文件，在此过程中绕过了由此带来的各种 ODR 问题。但通常你面临的是相反的任务：将非模块化代码转换为模块，以利用我们之前列出的优势。作为一个例子，我们将再次从 ExA_12 中的 ProperMath.h 开始。

```cpp
// ProperMath.h - 你的第一个、恰当的头文件
#ifndef PROPER_MATH_H
#define PROPER_MATH_H

auto square(const auto& x) { return x * x; } // 简写函数模板

const inline double lambda{ 1.303577269034296391257 }; // Conway常数

enum class Oddity { Even, Odd };
inline bool isOdd(int x) { return x % 2 != 0; }
inline auto getOddity(int x) { return isOdd(x) ? Oddity::Odd : Oddity::Even; }

#endif
```

你的第一个选择当然是继续努力，移除 `#include` 守卫，添加模块声明，在需要的地方添加 `export` 关键字，并重命名头文件。如果头文件与源文件配对，你当然也必须在那里添加匹配的模块声明。总的来说，对于大多数头文件来说，这是一个相当简单的过程。

虽然像那样重写你的代码肯定是一个可行的选择，但它仍然需要一些努力。你可能并不总是有时间一次性转换所有代码。此外，有时相同的源代码可能仍然需要用于其他你还不能使用模块的程序。在这种情况下，还有第二个选择。

语言因此允许你将大多数头文件作为模块的接口文件来编译并随后导入。你用来导入这样一个合成模块的一般语法如下所示：

```cpp
import "some/local_path/local_header.h";
import <some/external_path/external_header.h>;
```

也就是说，你可以不指定模块名，而是在 `import` 声明中指定头文件的名称，并用双引号或尖括号括起来。编译器解析这些路径名的方式将与预处理器解析 `#include` 指令的方式类似。

一个命名头文件的 `import` 声明，其行为就好像它导入了某个编译器生成的模块，该模块的接口是从该头文件中提取的。指定此接口的合成翻译单元被称为**头文件单元**。在这个头文件单元中，原始头文件中的所有声明都成为导出的。

你需要编译头文件单元的确切步骤是供应商特定的，因此你应该查阅你的编译器文档。

### 第45页

只有所谓的**可导入头文件**才能使用此语法进行编译和随后导入。具体什么构成可导入头文件又是实现定义的。然而，不可导入头文件的例子可能包括那些定义某些类型预处理宏的头文件。

即使是定义了基本宏的头文件也可能是可导入的。原因是，与常规模块不同，头文件单元确实导出在预处理阶段结束时定义的所有预处理宏定义。

> **警告**：头文件单元不会受到导入声明出现时定义的任何宏的影响。也就是说，在假设的 `import <Windows.h>;` 声明之前定义一个 `NOMINMAX` 宏，例如，是无效的。与所有模块接口单元一样，头文件单元只编译一次，因此你需要任何配置此单元的宏都应该在其被合成时定义。

如第2章所述，C++标准库的所有头文件，除了那些源自C标准库的头文件，都保证是可导入的。我们在整本书中一贯地利用了这一点。

> **提示**：大多数编译器都有一个选项，可以将某些 `#include <header>` 指令隐式地视为 `import <header>;` 声明。此选项通常至少对所有标准库头文件存在，但可能也对你自己或其他依赖项的头文件存在。因为这可以在不重写任何代码的情况下显著加快翻译速度，这可能值得研究。

### 第46页

**总结**

本附录讨论了多个主题，其中大部分在非模块化代码中特别重要。本章的要点包括：

-   **预处理阶段**执行指令，在编译前转换翻译单元中的源代码。所有指令处理完毕后，翻译单元将只包含 C++ 代码，不再有预处理指令。
-   你可以使用 `#define` 指令定义类对象或类函数的文本替换**宏**。在 C++ 中，你应该主要使用变量或函数模板来代替。
-   你可以使用**条件预处理指令**来控制哪些代码块进入翻译单元。如果相同的程序文件应该用不同的编译器编译，或者针对不同的目标平台，这尤其方便。
-   你可以使用 `#include` 指令将任何文件的全部内容插入到另一个文件中。你这样包含的文件几乎总是头文件。
-   `assert()` 宏使你能在**执行期间测试逻辑条件**，如果逻辑条件为假，则发出消息并中止程序。你可以使用 `static_assert()` 在编译时类似地检查条件。
-   **单一定义规则**规定，翻译单元中的每个实体只能有一个定义。如果整个程序中允许多个定义，它们仍然必须全部相同。然而，除非指定为 `inline`，否则不允许函数、成员函数或全局非 `const` 变量有多个定义。
-   一个名字可以具有**内部链接**、**外部链接**、**模块链接**，或者**没有链接**。
-   你使用**头文件**来包含其他头文件或源文件所需的定义和声明。头文件可以包含模板和类型定义、枚举、常量、函数声明、内联函数和变量定义以及命名命名空间。按照惯例，头文件名使用扩展名 `.h`。你的源文件通常将包含头文件中声明的所有非内联函数和变量的定义。
-   你可以使用 `#include` 守卫形式的条件预处理指令来确保头文件的内容在翻译单元内永远不会被复制。大多数编译器也支持更方便的 `#pragma once` 指令用于此目的。
-   在新代码中，你应该始终使用**模块**，现有代码应逐渐模块化。**头文件单元**——编译器基于遗留头文件生成的模块接口单元——可以在不重做现有代码的情况下帮助利用模块的优势。

## 主要内容提炼

这份附录详细介绍了在C++（特别是C++20之前）中管理多文件项目的底层机制，核心是**预处理、单一定义规则、链接**以及如何正确组织**头文件和源文件**。同时，它也强调了现代C++20**模块**相比传统方式的巨大优势。

### 核心概念

1.  **翻译过程**：将源代码转换为可执行文件经历三个阶段：**预处理** -> **编译** -> **链接**。预处理器的输出是**翻译单元**。
2.  **预处理指令**：以 `#` 开头，用于在编译前操作源代码。关键指令包括：
    -   `#define` / `#undef`：定义/取消定义宏。
    -   `#if` / `#ifdef` / `#ifndef` / `#else` / `#elif` / `#endif`：条件编译。
    -   `#include`：将指定文件内容插入当前位置。
    -   `#pragma`：提供编译器特定的功能。
3.  **宏**：危险的文本替换工具。**强烈建议**使用 `const` 变量、函数模板等C++特性代替宏，因为宏不进行类型检查、不遵守作用域、易产生副作用。
4.  **单一定义规则**：核心原则是每个实体（变量、函数、类等）在**一个翻译单元内只能定义一次**；对于常规（非内联）函数和变量，在**整个程序中也只能定义一次**。违反ODR会导致编译或链接错误。
5.  **链接**：决定了名字在不同翻译单元间的可见性。
    -   **外部链接**：可在整个程序的所有翻译单元中访问。
    -   **内部链接**：仅在当前翻译单元内访问。推荐使用**无名命名空间**（`namespace { ... }`）来实现，而非 `static` 关键字。
    -   **模块链接**：在同一个命名模块的翻译单元内访问。
    -   **无链接**：仅在当前作用域内访问（如局部变量）。
6.  **断言**：
    -   `assert()`：运行时断言，若条件为假则终止程序。常用于调试编程错误。可通过定义 `NDEBUG` 宏关闭。
    -   `static_assert()`：编译时断言，若条件为假则编译失败。用于检查编译时可确定的条件。

### 组织非模块化代码（头文件与源文件）

-   **头文件（.h）**：通常存放**声明**、类定义、模板定义、内联函数/变量定义、类型别名、枚举、常量等。
-   **源文件（.cpp）**：通常存放**定义**，特别是非内联函数和静态变量（包括类成员函数）的定义。

**关键技巧与注意事项**：

1.  **防止重复包含**：必须使用 **`#include` 守卫**或 `#pragma once` 来防止同一头文件在同一个翻译单元中被多次包含，从而避免ODR违规。
2.  **在头文件中定义函数/变量**：为避免程序范围内的ODR违规，头文件中定义的函数和变量（非 `const`）必须声明为 **`inline`**。这允许多个翻译单元有相同的内联定义。
3.  **类定义**：通常放在头文件中。成员函数可以在类内定义（隐式 `inline`），也可以在类外定义（需要显式 `inline` 才能放入头文件）。
4.  **模块导入**：**绝对不要**在头文件中使用 `import` 声明，这会破坏构建系统的依赖发现。

### 模块 vs. 头文件

模块（C++20）是现代C++推荐的代码组织方式，相比传统头文件有显著优势：

-   **更快的编译速度**：模块只处理一次，而头文件在每个包含它的翻译单元中都要重复处理。
-   **更好的隔离性**：模块的导入不影响宏定义，也不受外部宏影响，避免了命名污染和宏冲突。
-   **更清晰的结构**：不再需要 `#include` 守卫、`inline` 等繁琐的机制来避免ODR问题。
-   **头文件单元**：作为一种过渡手段，C++20允许将大多数头文件作为“头文件单元”导入（`import "header.h";`），从而在不重写代码的情况下享受部分模块化的好处。但可导入性有限制，且配置宏的方式不同。

总而言之，本附录是理解传统C++项目构建和避免常见链接错误的必读指南，并为转向现代C++模块提供了坚实的理论基础和迁移路径。
